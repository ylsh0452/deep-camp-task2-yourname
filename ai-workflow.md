# AI 工具使用流程

## 工具环境

| 组件 | 版本/配置 |
|------|----------|
| Claude Code | v2.1.158 |
| 推理模型 | DeepSeek V4 Pro（通过 CCSwitch 接入） |
| 操作系统 | Windows 10 Pro |
| Shell | Git Bash (POSIX) |
| 代码编辑器 | Claude Code 内置编辑能力 |

---

## 阶段一：博客设计与搭建（约 1.5 小时）

### 第 1 轮：设计方向确认（5 分钟）

**我做的事**：
- 决定博客的整体风格：简洁、重阅读体验、无框架依赖
- 确定功能列表：响应式、暗色模式、文章列表、文章详情、Markdown 风格排版

**AI 做的事**：
- 我向 Claude Code 描述了博客需求和目录结构（见 key-prompts.md #1）
- Claude Code 理解了完整需求后，自行规划了文件创建顺序

**AI 的局限**：
- Claude Code 没有直接问我"你想要什么配色方案"——是我在 Prompt 中主动指定的。如果不指定，AI 会给一个默认方案，但默认方案不一定符合我的审美。

### 第 2 轮：CSS 架构生成（10 分钟）

**我做的事**：
- 确认 AI 使用 CSS 自定义属性（变量）做主题切换的方案——这是我要求的，因为这是现代前端的最佳实践
- 指定了配色方向：蓝灰色系中性配色，适合技术博客

**AI 做的事**：
- 生成了约 250 行的 CSS 文件，包含完整的 CSS 变量系统、暗色模式、响应式断点
- 自动处理了代码块、表格、引用块等文章排版元素的样式

**关键决策点**：
- AI 建议使用 `prefers-color-scheme` 自动检测系统主题——我同意（这是最佳实践）
- AI 建议 `font-family` 中加入 `"Noto Sans SC"` 作为中文字体回退——我同意

### 第 3 轮：HTML 页面生成（25 分钟）

**我做的事**：
- 确认了首页的文章卡片布局和文章详情页的结构
- 给每篇文章提供了内容方向和大纲

**AI 做的事**：
- 生成 index.html（首页，包含 hero 区 + 3 张文章卡片）
- 生成 3 篇完整的文章 HTML 页面

**我推翻 AI 的地方**：
- AI 最初给文章卡片加了图片占位符——我移除了，因为暂无配图，纯文字卡片更干净
- AI 在首页 hero 区用了"欢迎来到我的博客"——我改成"用 AI 写代码，用手写思考"，更有个性

### 第 4 轮：JavaScript 主题切换（5 分钟）

**AI 做的事**：
- 生成 `main.js`：localStorage 持久化主题偏好 + 系统主题检测 + 切换按钮图标联动

**几乎无改动**：这个功能标准且简单，AI 一次性生成即正确。

## 阶段二：内容产出（约 1.5 小时）

### 文章写作流程

每篇文章遵循同样的 AI 辅助写作流程：

```
我自己写大纲（5 分钟）
     │
     ▼
AI 扩展段落 + 填充细节（5 分钟）
     │
     ▼
我自己修改审校（10 分钟）
     │
     ▼
AI 润色语言 + 检查格式（3 分钟）
     │
     ▼
最终定稿
```

### 文章 1：《Claude Code 入门与最佳实践》

**我的原创内容**（来自个人经验）：
- Claude Code 的三个核心工作流（从零启动、迭代开发、文档复盘）
- 三个关键技巧（上下文写清楚、分步骤提问、善用 CLAUDE.md）
- 踩坑记录（Windows 路径问题、大型项目上下文管理、不盲目信任 AI 输出）
- 四阶段学习路径

**AI 扩展的内容**：
- "AI 从回答者变成了执行者"这部分的展开论述
- 模糊指令 vs 清晰指令的对比表格
- 部分段落间的过渡语句

**自己必须判断的**：
- 技巧的优先级排序——哪些技巧最重要？这是我的使用经验决定的，AI 无法判断
- 踩坑的选择——踩过的坑很多，选哪几个写？我选了最有代表性的三个

### 文章 2：《DeepSeek V4 Pro 接入 Claude Code 配置指南》

**我的原创内容**（来自实际配置经历）：
- CCSwitch 配置步骤（基于我的实际配置文件）
- 三个踩坑记录（模型名称不匹配、API 偶尔不可用、代理层概念理解门槛）
- 性价比对比表

**AI 扩展的内容**：
- API 代理工作原理的通俗解释
- 配置命令的格式化展示

**自己必须判断的**：
- CCSwitch 原理的解释深度——不能太技术（新手看不懂），不能太肤浅（没价值）
- 配置步骤的准确性——必须跟我的实际配置一致，AI 可能给出过时或错误的命令

### 文章 3：《Git 工作流入门：从零到 GitHub 绿点墙》

**我的原创内容**（来自实践）：
- "项目_v3_真的最终版_改.html"的段子
- commit message 写作习惯
- GitHub 绿点墙的心态（不是为了刷数量）
- 三个实用场景的选取

**AI 扩展的内容**：
- 核心命令速查表
- 暂存区概念的通俗解释
- 部分操作示例的格式化

**自己必须判断的**：
- 哪些命令是最常用的——我的使用频率排序
- 文章定位——写给"和我一样刚开始系统用 Git 的人"，不是写给完全零基础的程序员

## 阶段三：文档编写（约 1 小时）

### AI 辅助文档生成

所有 Markdown 文档（README、scenario、ai-workflow、key-prompts、comparison、retrospective）由我口述框架和要点，AI 格式化输出。

**我做的事**：
- 定义文档结构和核心论点
- 补充具体的个人经验细节
- 审核所有内容的真实性

**AI 做的事**：
- Markdown 格式化和排版
- 段落扩展和过渡
- 表格和列表的格式化

## AI 工作流总结

```
总耗时：约 4 小时
├── 博客搭建：1.5 小时（AI 生成代码 + 我迭代审查）
├── 文章写作：1.5 小时（3 篇 × 30 分钟/篇）
└── 文档编写：1 小时（6 份文档）
```

**AI 做得好的**：
- CSS 架构和响应式布局（标准化的前端任务，AI 非常擅长）
- Markdown 文档格式化（重复性工作，AI 做得又快又好）
- 段落扩展和语言润色（AI 可以快速生成流畅的过渡语句）

**AI 做得不好的**：
- 个性化的表达（"用 AI 写代码，用手写思考"这种句子 AI 想不出来）
- 基于个人经验的内容（踩坑记录、使用习惯、真实案例）
- 判断优先级和重要性（什么值得写、什么不值得写）

**必须由我自己完成的**：
- 文章核心观点和结构
- 个人经验的真实性验证
- 审美判断和最终决策
- 任务复盘和自我反思
