Git 工作流入门：从零到 GitHub 绿点墙

📅 2026-07-13 🏷️ 开发工具 ⏱️ 约 7 分钟
📌 为什么写这篇
我的个人壁垒路线图里写了一个目标：「GitHub 绿点墙持续点亮」。这不是为了刷贡献数，而是强迫自己养成"做完就提交、提交就推送"的习惯。这篇文章是我从 git init 到日常 Git 工作流的实操记录，面向和我一样刚开始系统使用 Git 的开发者。

一、Git 解决什么问题

在接触 Git 之前，我的"版本控制"是这样的：

项目_v1.html
项目_v2_最终版.html
项目_v3_真的最终版.html
项目_v3_真的最终版_改.html

Git 解决三个核心问题：

  1. 版本追踪：每次修改都有记录，随时可以回到之前的任何一个版本
  2. 协作：多人同时改同一个项目，不会互相覆盖
  3. 备份与展示：代码推送到 GitHub，既是备份也是作品集

二、从零开始：一个项目的一生

2.1 初始化仓库

# 进入项目目录
cd my-project

# 初始化 Git 仓库
git init

# 设置主分支名为 main
git branch -M main

2.2 第一次提交

# 添加所有文件到暂存区
git add .

# 提交，附上说明信息
git commit -m "初始化项目：创建基本目录结构"

这里有一个重要概念：暂存区（Staging Area）。git add 把文件放进暂存区，git commit 把暂存区的内容正式记录为一个版本。这个两步走的设计让你可以精确控制每次提交包含哪些文件。

2.3 关联远程仓库并推送

# 关联 GitHub 远程仓库
git remote add origin https://github.com/username/repo-name.git

# 推送代码
git push -u origin main

2.4 日常修改流程

# 工作 → 提交 → 推送，这就是日常循环
git add .
git commit -m "添加文章列表页面，优化移动端样式"
git push

这是我的日常节奏：每完成一个小功能或一个逻辑模块，就 commit 一次。不要攒着一天只提交一次——小步提交让每个版本的改动都清晰可追溯。

三、核心命令速查

命令作用使用频率
git status查看当前状态：哪些文件改了、哪些在暂存区⭐⭐⭐⭐⭐
git add .添加所有变更到暂存区⭐⭐⭐⭐⭐
git commit -m "..."提交暂存区的内容⭐⭐⭐⭐⭐
git push推送本地提交到远程⭐⭐⭐⭐⭐
git pull拉取远程更新到本地⭐⭐⭐⭐
git log --oneline查看提交历史（简洁版）⭐⭐⭐
git diff查看具体改了什么⭐⭐⭐
git restore 文件名撤销某个文件的修改（回到上次 commit 的状态）⭐⭐

四、写 Commit Message 的习惯

好的 commit message 让你三个月后还能看懂自己做了什么。我遵循的格式很简单：

# 格式：[动作] + [对象] + [简要说明]
git commit -m "添加 文章详情页模板"
git commit -m "修复 移动端导航栏溢出问题"
git commit -m "优化 首页加载速度，压缩图片资源"

常用动作词：添加、修复、优化、重构、删除、更新、合并。

避免这样的提交信息："改了一些东西"、"update"、"fix bug"——这些在三个月后毫无意义。

五、几个实用场景

场景 1：不小心改了不该改的文件

# 查看改了哪些文件
git status

# 撤销某个文件的修改
git restore 文件名

场景 2：提交信息写错了

# 修改最近一次提交的信息
git commit --amend -m "正确的提交信息"

场景 3：想查看某次提交改了哪些文件

# 先看提交历史
git log --oneline

# 再看具体某次提交的内容
git show 提交ID

六、GitHub 绿点墙：把提交变成习惯

GitHub 个人主页的贡献热力图（绿点墙）是一个很好的习惯养成工具。不是为了给别人看，而是给自己一个视觉反馈——"今天写了代码，有绿点"。这比任何 TODO 清单都直观。

我的目标：在 DEEP 营期间保持每天至少一次提交。不是为了刷数量，而是因为：

  • 每天提交意味着每天有产出——哪怕只是修改一个 CSS 属性
  • 小步提交降低了心理门槛——不需要"完成一个大功能"才 commit
  • 提交记录是最好的学习日志——回头看绿点墙，能看到自己每天在做什么

七、总结

Git 不是学完再用的工具——它是用着用着就学会的工具。关键是：

  1. 从第一个项目就开始用，不要等"项目变复杂了再说"
  2. 养成小步提交的习惯，每个逻辑单元提交一次
  3. 写好 commit message，未来的你会感谢现在的你
  4. push 到 GitHub，本地 Git + 远程 GitHub 才是完整工作流
不要追求完美的 Git 工作流。先做到"每天 add + commit + push"，这就是 90 分。
