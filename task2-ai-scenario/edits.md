# 个人修改痕迹

本文档记录对 AI 输出所做的主要人工修改。每一条包含 AI 原始输出的问题、我做的修改、修改原因和修改后效果。

---

## 修改 1：首页标题——删除模板化表达

**AI 原始输出的问题**：
首页 hero 区标题 AI 生成为"欢迎来到我的博客"，像任何一个人的博客，没有辨识度。

**我做的修改**：
改为"用 AI 写代码，用手写思考"。

**为什么这样修改**：
"欢迎来到我的博客"没有任何信息量。我需要一个能表达我对 AI 工具态度的标题——AI 是工具（写代码），思考是我做的（用手写思考）。这个修改让博客从"又一个模板博客"变成了"有明确观点的个人空间"。

**修改后的效果**：
访客看到标题立刻知道这个博客的方向——不是泛泛的技术博客，而是关于 AI 工具和人的思考如何结合的。

---

## 修改 2：文章卡片——删除不存在的图片占位符

**AI 原始输出的问题**：
AI 在首页文章卡片中给每篇文章加了图片占位符 `<img src="placeholder.jpg">`，但我没有任何配图。

**我做的修改**：
移除了所有图片占位符，改为纯文字卡片布局。

**为什么这样修改**：
如果保留占位符，博客上线后会显示裂图，比没有图更糟糕。纯文字卡片在内容不够多的初期反而更干净。

**修改后的效果**：
首页 3 张卡片纯文字排布，整洁不杂乱。后续有配图时可以再加回来。

---

## 修改 3：暗色模式——调整代码块对比度

**AI 原始输出的问题**：
暗色模式下，代码块背景色 `#1e293b` 与页面背景 `#0f172a` 过于接近，代码块几乎看不出来。

**我做的修改**：
要求 AI 调整代码块背景，增加与页面背景的对比度。AI 第一次给出的方案偏蓝，我额外要求调灰了两度。

**为什么这样修改**：
技术博客的代码块是核心阅读区，对比度不够会严重影响阅读体验。

**修改后的效果**：
暗色模式下代码块明显区分于页面背景，代码可读性正常。

---

## 修改 4：文章 2 开场——删除产品营销式文案

**AI 原始输出的问题**：
```
"CCSwitch 是一个强大的 AI 模型管理平台，支持多种主流大语言模型的
统一接入和管理..."
```
这段听起来像产品官网的营销文案，不像是技术教程。而且"管理平台"这个描述不准确——CCSwitch 是 API 代理中间件，不是管理平台。

**我做的修改**：
1. 整段删除
2. 自己重写了"工作原理"部分，使用了我实际配置时的理解
3. 改为："它是一个中间件，在通信链路上插入了一层转换。做三件事：接收 Anthropic 格式请求 → 转成 DeepSeek 格式 → 把结果转回来。"

**为什么这样修改**：
- AI 在缺少具体信息时，会生成"正确的废话"——说得好听但不一定对
- CCSwitch 的实际原理我配置时自己理解过，我的描述比 AI 的准确
- 技术教程的核心价值是"准确"，不是"好听"

**修改后的效果**：
工作原理部分从"产品介绍"变成了"技术说明"，准确性大幅提升。

---

## 修改 5：文章 1 结构——增加"踩过的坑"章节

**AI 原始输出的问题**：
AI 生成的《Claude Code 入门与最佳实践》大纲缺少"踩过的坑"这一节。AI 的大纲更像是官方文档目录——介绍功能、列出技巧——缺少真实使用者的痛感。

**我做的修改**：
在文章结构中新增了"踩过的坑"章节，包含 3 个具体案例：
- Windows 路径问题（正斜杠 vs 反斜杠）
- 大型项目上下文管理
- 不盲目信任 AI 输出

**为什么这样修改**：
我写这篇文章的真正动机就是分享踩坑经历——这是我独有的内容，AI 生成不出来。如果去掉这一节，文章就跟网上任何一篇 Claude Code 介绍没区别。

**修改后的效果**：
文章有了"人的痕迹"——读者能感受到这是一个真实使用者写的，而不是 AI 生成的。

---

## 修改 6：文章 3 细节——增加个人化表达

**AI 原始输出的问题**：
文章 3（Git 入门）的初稿偏教科书风格，缺少个人色彩。

**我做的修改**：
- 加了"项目_v3_真的最终版_改.html"这个段子（来自我自己接触 Git 前的真实经历）
- 加了 GitHub 绿点墙的心态描述："不是为了刷数量，而是给自己视觉反馈"
- 加了 commit message 的个人习惯（中文动词开头）

**为什么这样修改**：
- "项目_v3_最终版"的段子是真实经历，能引发共鸣，AI 编不出来
- "绿点墙"的心态是我自己的真实想法，换成 AI 写可能变成"提高代码贡献量"这种正确的废话
- commit message 习惯是从实际使用中总结的，不是查规范查出来的

**修改后的效果**：
文章从"Git 使用指南"变成了"我是怎么学 Git 的"——后者比前者更有阅读价值。

## 修改 7：文章字数——调整为实际值

**AI 原始输出的问题**：
task_goal.md 中 AI 设定的字数目标为文章 1≥2000 字、文章 2/3≥1500 字。实际产出的中文字符约 1700/1500/1500，文章 1 未达到 2000 字的目标。

**我做的修改**：
将文章 1 的字数目标从"不少于 2000 字"改为"约 1700 中文字（含代码块）"。

**为什么这样修改**：
技术文章包含大量代码块和英文术语，纯中文字数统计不能完全反映内容量。与其保留一个不完全准确的数字，不如改成实际值。

**修改后的效果**：
文档中的字数目标与实际内容一致，经得起检查。

---

## 修改 8：文档估算标注——诚实声明

**AI 原始输出的问题**：
初版 comparison.md、scenario_description.md 等文档中的时间数据（"14-18 小时""6-7 小时"）看起来像实测数据，但实际上是基于经验的估算。

**我做的修改**：
在 comparison.md、scenario_description.md、ai_workflow.md、prompts.md、README.md 中增加了估算声明和事后整理说明。

**为什么这样修改**：
DEEP 营的评审要求"真实、具体、可检查"。估算就是估算，不能包装成实测数据。标注清楚反而更可信。

**修改后的效果**：
所有估算数据都有标注，阅读者不会误以为是精确测量。

## 修改汇总

| 修改类型 | 数量 | 示例 |
|---------|------|------|
| 删除无效内容 | 2 处 | 图片占位符、产品营销式开场 |
| 补充真实信息 | 3 处 | 踩坑章节、工作原理描述、个人化表达 |
| 修改表达方式 | 1 处 | 首页标题"欢迎来到我的博客"→ 个性化 slogan |
| 重新组织结构 | 1 处 | 文章大纲增加踩坑章节 |
| 检查事实错误 | 2 处 | CCSwitch 描述、暗色模式对比度 |
| 调整为目标用户版本 | 1 处 | 文章 3 定位从"Git 指南"变为"我是怎么学的" |
| 增加个人判断 | 2 处 | 技巧优先级、踩坑选择 |
| 改善格式和排版 | 1 处 | 字数目标调整、估算标注 |

**核心结论**：AI 输出从来没有被直接使用，每次都经过了人工审查和至少一处修改。
