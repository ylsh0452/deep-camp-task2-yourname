# 真实场景说明

## 场景类型

**内容表达场景** — 个人技术博客搭建与原创文章产出。

---

## 1. 这个场景具体是什么？

我需要在互联网上有一个属于自己的技术博客网站，并且产出第一批 3 篇原创技术文章。

具体来说：
- 从零搭建一个静态博客网站（HTML/CSS/JS，不依赖框架）
- 支持响应式布局（桌面 + 手机）和暗色模式
- 写 3 篇技术文章：《Claude Code 入门与最佳实践》《DeepSeek V4 Pro 接入 Claude Code 配置指南》《Git 工作流入门：从零到 GitHub 绿点墙》
- 部署到 GitHub Pages，生成一个公开可访问的链接

---

## 2. 这个问题真实存在在哪里？

存在我任务一写的《个人壁垒路线图》里：

> 阶段二 项目实战期（2026.10 — 2027.06）  
> 目标：建立技术输出，写技术笔记/博客/复盘，累计 10+ 篇

这个目标从 7 月 10 日写完路线图之后，一直没有启动。原因不是"不想写"，而是两个阻力叠加：
- **技术阻力**：没有一个现成的博客网站可以发布文章
- **心理阻力**：第一次写技术文章，不知道写什么、写到什么程度算合格

所以这个问题真实存在于我自己的计划里——它是写进路线图但尚未执行的事项。同时它也存在于 DEEP 营的任务体系里：任务二要求交付一个真实场景的完整产出物。

---

## 3. 是我自己的问题，还是别人也可能遇到？

**核心是我自己的问题**。我路线图上写了、还没开始做，任务二是推动我启动的最佳时机。

但这个问题也有共性——参加 DEEP 营的学员或多或少都面临"技术输出"的需求。任务要求提交 GitHub 仓库，意味着每个人的产出都要接受同伴评审。我选择博客作为场景，交付物对其他学员也有参考价值：他们可以看到一个完整的 AI 辅助博客搭建 + 写作流程是什么样的。

---

## 4. 不用 AI 时，我原本会怎么解决？

不用 AI 的话，完成同样的任务，我的流程是：

| 步骤 | 内容 | 预估时间 |
|------|------|---------|
| 1 | 手写 HTML/CSS 搭建博客框架 | 3-4 小时 |
| 2 | 在浏览器和编辑器之间反复切换调试样式 | 1-2 小时 |
| 3 | 手动适配响应式布局 | 30-60 分钟 |
| 4 | 手动实现暗色模式切换 | 30-60 分钟 |
| 5 | 手写 3 篇技术文章（每篇 2-3 小时） | 6-8 小时 |
| **合计** | | **12-16 小时** |

实际操作中还有一个隐藏成本：因为工作量大，我很可能会延迟启动——"等周末有大块时间再说"，然后周末被其他事情占满，继续推迟。

---

## 5. 不用 AI 时，困难在哪里？

困难不在技术难度本身（HTML/CSS 我都会写），而在于**效率瓶颈**和**启动门槛**：

**效率瓶颈**：
- 写 CSS 最耗时的是"试颜色值"——改一个十六进制、刷新浏览器、不满意、再改。AI 可以一次性给出一套和谐的配色方案（CSS 变量体系），直接跳过这个循环
- 文章写作最耗时的是"发呆"——知道要写什么主题，但面对空白编辑器不知道怎么起头。AI 可以先给一个段落草稿，我在上面修改，比从零写快得多

**启动门槛**：
- 手写 12-16 小时意味着分散到多个周末才能完成。在"完成全部 DEEP 营任务"的节奏下，很难为博客单独留出这么大块的时间
- AI 把总时间压缩到约 4 小时，一个下午就能做完，启动门槛从"拆成三个周末"变成了"今天下午搞定"

**质量稳定性**：
- 手写文章时，第三篇的质量通常比第一篇差（疲劳效应）。AI 辅助写作让每篇的质量更均匀，因为 AI 不会累

---

## 6. 这个场景为什么适合作为 AI 应用实验？

这个场景天然包含了 AI 工具可以发挥作用的所有环节，同时保留了必须由人来判断的核心部分：

**AI 擅长的（这个场景可以验证）**：
- 代码生成（CSS 架构、HTML 结构、JS 交互）
- 格式化和排版（Markdown 文档、表格、列表）
- 段落扩展和语言润色

**AI 不擅长的（这个场景可以暴露局限）**：
- 个性化表达（"用 AI 写代码，用手写思考"这个 slogan 是我想的，AI 想不出来）
- 基于个人经验的内容（踩坑记录、使用习惯、真实案例，AI 没有这些经历）
- 审美判断（配色好不好看、排版是否舒服）
- 优先级判断（哪些技巧最重要？哪些踩坑值得写？）

**总之人机分工很清晰**：AI 做执行（写代码、扩段落、排格式），人做判断（定方向、审内容、做决策）。这个边界清晰，所以适合作为实验场景。
