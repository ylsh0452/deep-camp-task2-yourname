# AI 工具选择与分工

## 使用的 AI 工具

本次任务使用 2 个 AI 工具协同工作：Claude Code 作为开发环境和任务编排层，DeepSeek V4 Pro 作为底层推理引擎。

| 工具名称 | 负责环节 | 为什么选择它 | 输入材料 | 输出结果 | 存在的问题 |
|---------|---------|-------------|---------|---------|-----------|
| Claude Code (v2.1.158) | 全流程任务编排：目录创建、代码生成、文件管理、Git 操作、文档编写 | 不只是聊天工具，而是完整的开发环境。能直接操作文件系统、执行 shell 命令，一次对话完成多文件项目的搭建。相比在 ChatGPT 里复制粘贴代码，效率差距巨大 | 博客功能需求描述、目录结构规划、设计风格要求 | CSS 样式文件（250+ 行）、5 个 HTML 页面、JS 主题切换脚本、6 份 Markdown 文档 | 生成的默认配色需要 2-3 轮微调才能达到满意效果；偶尔在 Windows 路径处理上有兼容问题（需手动用正斜杠） |
| DeepSeek V4 Pro | 底层推理引擎：代码逻辑生成、文章段落扩展、语言润色、中文内容产出 | 通过 CCSwitch 接入 Claude Code 作为后端模型。中文能力突出，文章写作流畅自然；API 成本低于原生 Claude 模型，适合日常高频使用 | Claude Code 转发的所有任务请求（开发需求、写作指令、润色要求） | 所有代码和文章的原始生成内容，经由 Claude Code 呈现给用户 | 服务偶尔不可用导致请求超时；推理速度在高峰期较慢；对英文技术术语的生成不如原生 Claude 模型精确 |
| Microsoft Edge (headless) | 博客页面自动截图 | 不是 AI 工具，是辅助工具。Windows 自带，无需额外安装，支持 headless 模式命令行截图，用于快速生成过程截图 | 本地 HTML 文件路径 `file:///D:/...` | 6 张 PNG 截图（首页、文章页、移动端等） | 暗色模式截图无法通过 CLI 参数直接切换（需 JS 执行）；只能截取静态页面初始状态 |

## 为什么这样分工

```
用户指令
   │
   ▼
Claude Code（编排层）
   │  ├── 理解需求，规划执行顺序
   │  ├── 操作文件系统（创建/修改/删除）
   │  ├── 执行 shell 命令（git / mkdir / curl）
   │  └── 调用推理引擎 ↓
   │
   ▼
DeepSeek V4 Pro（推理层）
   │  ├── 代码生成（HTML / CSS / JS）
   │  ├── 文章段落扩展与润色
   │  └── 文档格式化与排版
```

- **Claude Code** 负责"怎么做"——它把用户需求拆成可执行的步骤，操作文件、执行命令、管理整个项目的生命周期。
- **DeepSeek V4 Pro** 负责"做什么内容"——所有的代码、文章、文档的实际生成都由它完成。

两者不是"多装了几个工具"的关系，而是**分层协作**：编排层只管流程，推理层只管产出。如果只用 DeepSeek 的 Web 聊天界面，就无法直接操作文件和执行命令；如果只用 Claude Code 的原生模型，中文文章质量和 API 成本就不如 DeepSeek。

## 没有盲目堆工具

本次任务只用了一套主力工具链（Claude Code + DeepSeek V4 Pro），而不是罗列多个工具凑数。选择标准是：

- **每个工具解决了不可替代的问题**：Claude Code 管流程，DeepSeek 管生成，缺一个就完成不了
- **不为了"多用工具"而引入新工具**：没加 ChatGPT、没加 Cursor、没加 Copilot——不是因为它们不好，而是当前工具链已经够用。工具数量不是目的，效率才是
